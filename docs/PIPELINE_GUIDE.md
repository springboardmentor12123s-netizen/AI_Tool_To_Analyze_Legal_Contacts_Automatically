# ClauseAI Pipeline Guide

This document explains the internal Multi-Document Asynchronous Processing Pipeline for ClauseAI.

## Overview
ClauseAI's data pipeline is built for high concurrency. By migrating from sequential processing to an asynchronous architecture using `asyncio` and semaphores, the system can parse multiple huge contracts in parallel without hitting LLM rate limits.

## The Multi-Document Flow

### 1. Ingestion (`src/bulk_processor.py`)
- User uploads multiple documents via the `/analyze_bulk` API.
- The `bulk_processor.process_all_documents` creates an asynchronous task for each document.
- An `asyncio.Semaphore(MAX_CONCURRENT=5)` limits concurrent document processing to prevent overwhelming the Groq LLM API.

### 2. Caching
- A SHA-256 hash is generated from the combined document text and the `ReportConfig`.
- A lookup occurs in `data/cache/*.json` using `DocumentCache`.
- If a hash match is found, the system immediately returns the cached result, saving LLM execution time.

### 3. Agent Graph Extraction (`src/agent_graph.py`)
- The document text is loaded and chunked into candidate clauses.
- **Parallel Extraction**: Domains (Compliance, Financial, Legal, Operations) run concurrently across clauses via `ParallelExtractor` (`src/parallel_extractor.py`).
- **Pinecone Store**: Intermediate clause objects, embedded with their detected metadata (risk level, exposure etc.), are written to the local Pinecone Vector database.

### 4. Report Generation
- The `Synthesis Node` retrieves the relevant findings across all domains.
- A customized prompt template is generated by the `ReportGenerator` using user parameters (Tone, Structure, Focus).
- The LLM synthesizes everything into the final `AgentState['final_report']`.
